V2 Phase 2
1. Replace all deprecated PyTeal code and build the core on-chain smart contracts using modern AlgoKit tooling.
Create a new /contracts folder and rewrite the insight listing, escrow, and reputation logic entirely in Python using the Algorand Python SDK and AlgoKit templates; define each trading insight as an ASA that stores price, seller wallet, and IPFS hash; add a simple escrow mechanism that releases content access only after x402 payment confirmation; include a basic stateful app for on-chain reputation scoring; then compile, test locally with goal or pytest, and prepare for deployment so the blockchain layer is production-ready and fully compliant with current Algorand standards.


To-Do:
1. Create the dedicated contracts folder inside the backend.
Inside your project root, run mkdir backend/contracts so all smart-contract files live together and are automatically picked up by AlgoKit build commands.
2. Generate a fresh smart-contract template using the official AlgoKit generator.
Run algokit generate smart-contract --name InsightListing (and repeat for Escrow and Reputation if the generator prompts) — this creates ready-made contract.py files with the correct ARC4Contract boilerplate, approval/clear programs, and .algokit configuration.
3. Install the Algorand Python compiler dependencies (if not already present).
In your activated venv run pip install algorand-python puya (this gives you algopy for writing contracts and the Puya compiler that replaced PyTeal).
4. Rewrite the InsightListing contract in Python using native ARC4 syntax.
Open backend/contracts/insight_listing.py, replace everything with a proper class InsightListing(ARC4Contract) that has an @arc4.abimethod for create_listing(price: arc4.UInt64, seller: arc4.Address, ipfs_hash: arc4.String); store the three values (price, seller wallet, IPFS hash) in global state or a Box for each unique listing ID.
7. Configure the contracts in algokit.toml for automatic compilation.
Edit algokit.toml (or the contracts section) to list all three contracts under [contracts] so algokit project run build will compile them together.
8. Compile all three contracts to TEAL using the modern AlgoKit command.
5. Add the escrow logic as a separate contract or combined method.
In backend/contracts/escrow.py (or inside the same file if you prefer one contract), implement an @arc4.abimethod called release_after_payment(buyer: arc4.Address, listing_id: arc4.UInt64) that checks the x402 payment transaction group and only then allows the buyer to claim the content unlock by updating state.
6. Build the basic stateful reputation scoring contract.
In backend/contracts/reputation.py create a simple ARC4Contract with methods update_score(seller: arc4.Address, new_score: arc4.UInt64) and get_score(seller: arc4.Address) -> arc4.UInt64; store scores in global state or a Box map so agents can read reputation on-chain before buying.
Run algokit compile py (or algokit project run build) from the project root — you should now see clean approval.teal and clear.teal files generated for each contract with zero PyTeal warnings.
9. Test the contracts locally with AlgoKit’s built-in tester.
Run algokit project run test (or pytest backend/contracts if you added simple tests) using the localnet sandbox; simulate a full listing → payment → escrow release → reputation update cycle to confirm every method works and state updates correctly.
10. Prepare the contracts for Testnet deployment by generating the application specification.
Run algokit generate client (or algokit project run build) again to produce the application.json (ARC-32) files for each contract; copy the App IDs placeholders into your .env so the FastAPI backend and LangChain agent can reference them later — your blockchain layer is now fully modern, compliant, and ready for deployment in the next roadmap point.
