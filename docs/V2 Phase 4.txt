V2 Phase 4
1. Integrate IPFS storage for insight content with secure pinning and on-chain referencing.
Sign up for a free Pinata account, generate the JWT, and create reusable helper functions that upload text files, return the CID, and immediately write that CID into the on-chain ASA during listing; add automatic pinning logic and fallback error handling so that every human-uploaded insight is permanently stored off-chain yet verifiably linked on Algorand for instant retrieval after payment.




To-Do:
1. Sign up for a free Pinata account.
Go to https://app.pinata.cloud/register, create a new account using your email (or GitHub), verify your email, and log in — this gives you immediate access to the free tier (1 GB storage + unlimited pins, perfect for hackathon insights).
2. Generate a dedicated API JWT token on Pinata.
In the Pinata dashboard go to “API Keys” → “New Key” → name it “mercator-insights” → enable only “pinFileToIPFS” and “unpin” permissions → copy the generated JWT string (it starts with “eyJ...”).
3. Add the Pinata JWT securely to your environment file.
Open .env (or .env.testnet if you prefer separation) and add the line PINATA_JWT=eyJhbGciOi...your-full-jwt-here... exactly as copied — never commit this file to Git.
4. Create the IPFS helper module in the backend.
Inside backend/ run mkdir -p utils then create a new file backend/utils/ipfs.py — this will hold all reusable upload and pinning logic.
5. Install the IPFS HTTP client (if not already present).
In your activated venv run pip install ipfshttpclient to ensure the library is available for making authenticated calls to Pinata’s gateway.
6. Write the core upload_and_pin function with automatic pinning.
In ipfs.py create an async def upload_insight_to_ipfs(text: str, filename: str = "insight.txt") that converts the text to bytes, calls Pinata’s /pinning/pinFileToIPFS endpoint using the JWT, automatically pins the file, and returns the full CID (starting with “Qm...”).
7. Add robust fallback error handling and retry logic.
Wrap the Pinata call in try/except with a 3-second retry on timeout or 429 errors; if it fails completely, raise a clear IPFSUploadError with a user-friendly message (“Could not store insight on IPFS — please try again”) so the seller flow never hangs.
8. Create a helper to link the CID to the on-chain ASA.
In the same ipfs.py add a function store_cid_in_listing(cid: str, listing_app_id: int, seller_address: str) that uses the py-algorand-sdk + the already-deployed InsightListing contract client to call the create_listing method, passing price, seller wallet, and the new IPFS CID as parameters.
9. Test the IPFS helper functions locally with a sample insight.
Create a quick test script backend/utils/test_ipfs.py that imports the functions, calls upload_insight_to_ipfs("Sample trading insight: Buy NIFTY above 24500..."), prints the returned CID, then calls the store function using your TestNet App ID — run it with python -m backend.utils.test_ipfs and verify the CID appears correctly on the Algorand TestNet block explorer.
10. Export the helper so the FastAPI backend can import it easily.
In backend/utils/__init__.py add from .ipfs import upload_insight_to_ipfs, store_cid_in_listing and in backend/main.py (or wherever your FastAPI app lives) add from utils.ipfs import upload_insight_to_ipfs so the helper is ready to be called from the /list endpoint in the next roadmap point — your IPFS integration is now complete, secure, and fully linked to the live on-chain ASA.
